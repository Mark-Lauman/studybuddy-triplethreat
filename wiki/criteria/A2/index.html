Assignment 2: Requirements and Initial Design

This assignment has two main parts: a requirements document, and some initial design documents.

Requirements Document

Your requirements document will be written in the form of a user manual. It might seem strange to write a user manual before you have written any code for the system, but it is a good way to focus on the user-level requirements without yet going into implementation detail.

While we call this a user manual, this is in fact a combination user manual (meant for end-users) and requirements document (meant for system designers). You must therefore write this in a language and format that is useful for both end-users and designers: don't make it too technical, nor too vague.

At this early point in the project it is more important to make this user manual useful for designers. Make sure there is enough detail and information in it so that a system designer could ideally design and implement your system from the manual alone. A polished user-friendly version of this manual can be created for end-users later on.

Structure your user manual as follows:

  • Introduction This explains what the system does and what is useful for.
  • Intended Audience List the kinds of users for this system in some detail (certainly more detail than in the Vision and Scope document from the previous assignment). Make clear what experience, expertise, and goals you assume they have.
  • Features Catalog every significant feature of the system. This is should be an exhaustive list. Hint: You may want to give each of these features its own anchor or page, so that in the design document (described below) you can easily refer to them.
  • Non-functional requirements Describe your system's non-functional requirements. Hint: Look at the table on page 125 of Sommerville.
  • Example Tutorials Provide 3 - 5 (or more, if necessary) examples of typical usage scenarios. These need to be highly detailed, step-by-step tutorial-like descriptions of how to use your system's major features. You must include screenshots of how your system will behave (users don't want to read through a lot of text!).
  • Glossary List all technical terms and their precise definitions here. These will be terms about your application, not about programming or software engineering. For example, in a grade-keeper application, terms like "course", "grade", "mark", and "GPA" all need precise definitions.

Design Document

Your design document outlines the high-level framework of your system, and it is what the programmers will refer to during implementation.

Your design document should have these sections:

  • Guidelines List all technical guidelines (e.g. "Development will be done using the Eclipse IDE"), and any relevant ethical or legal issues.

  • System Diagrams This should include 2 or 3 (or more, if needed) substantial diagrams showing the major modules, classes, and functions of your system relate and share data. Important: these must be diagrams as described in chapter 8 (System models) of the textbook; you must follow the proper rules and notations for the diagrams you use. It is not acceptable to make up your own kind of diagram! Hint: If you are using an object-oriented programming language, than one of your diagrams should be a class diagram.

  • Data Requirements This is a summary of all the I/O for your system, including exact definitions of file formats, what other systems (if any) yours interacts with, and how the user interacts with the system via the mouse/keyboard/wii-mote/etc.

  • Feature Priority You will be going through three implementation iterations during the course of this project --- version 1, version 2, and version 3. Label each feature listed in your user manual, as being delivered in version 1, 2, or 3. Hint: An ordered list of hyperlinks to the relevant features in the user manual might be a good way to organize this.

    Remember that your Vision and Scope document from the previous assignment should have a similar (but less detailed) list of what features are planned for each version; so that would be a good starting point.

    It is important that version 1 (and, of course, the following versions) be a usable system. Version 1 should implement all your promised version 1 features completely and correctly! Also keep in mind that version 3 is the final "gold" version of your project, and anything not implemented by version 3 will not be implemented at all (at least during this course!).

    Think hard about a good ordering for your features, taking into account things like their dependencies and complexity. Do not over-promise (too many features) or under-deliver (too few features).

Tagging Your Assignment Submissions

Using your version control system, please save copies of your assignment files exactly as they were submitted. This means all submitted files: source code, documentation, diagrams, wiki pages --- whatever files you have in your assignment. In Subversion, all you need to do is copy your final assignment files into a folder named something like "a2submittted". Since Subversion repositories use "cheap copying", making a copy of an entire folder structure is sometimes called tagging, since in reality no files are copied.

Also, please provide explicit instructions for how to check out these specific versions of your assignment so there is no confusion about which folder is the one containing the officially submitted files.

Of course, you can (and should!) change and update your previous assignments as necessary. Use a sensible naming/numbering scheme to keep track of updates, e.g. for each assignment, tag each previous assignment to take into account updated files.

However you exactly do this tagging, make it easy to understand --- we should not have to do anything more than look at folder names to clearly and completely understand your tagging scheme.
