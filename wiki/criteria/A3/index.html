Assignment 3: Version 1

The goal for this assignment is to produce a working version of your system that implements all your version 1 features.

In addition to the working system, your team must also do the following.

Update the User Manual and Design Document

Prepare new versions of the user manual and the design document so they are consistent with the system you actually implement. The reason for this that your next version of the system will be built from this one, so it is essential to have up to date and accurate documentation.

Do not delete your old user manual and design document: keep all the files for your previous assignments so that you can see how your system has changed over time.

Do not change or rewrite your vision and scope document (unless requested by the marker!). It is instructive to see how your original ideas about the system compare to the working version you actually create.

Hint: If your user manuals are text documents, then if you keep them in your Subversion repository it will be easy to keep old versions and list the precise differences between them.

Source code documentation

Source code documentation is an essential part of maintainable software, and so your team is expected to produce high-quality source code.

All source files must have a standard header that names the file, lists the programmers who have worked on it, gives the team name, etc.

Your group must follow a coding standard with all of its files. This coding standard must be explicitly written down. For example, identifiers (i.e. names of variables, classes, methods, and files) should be self-descriptive, and code formatting should be perfect and consistent.

All functions, classes, and files should have source code comments that explain their purpose, with examples if needed. Generally, the more important or complicated the function, the more detailed the comments.

Automatic Unit Tests

Testing is an essential part of any engineering discipline, and in this course you must not only test your software, you must prove that you have tested it by providing the test cases you performed. Untested code will be considered incomplete: it is your responsibility to prove your system works as claimed!

All your implemented code must be tested using automatic unit tests. This can be as simple as using assert statements to test key functions, or you could use a tool such as JUnit? (for Java) or doctest (for Python).

For code that is difficult to write automatic tests for (e.g. GUI testing), then provided a written description of how it was tested. This must be in enough detail to re-create the testing you did.

Usability

The interface should be easy to use and appealing, and your program should not crash, nor claim to have any features that do not currently work. Keep in mind that fancy-looking graphics do not guarantee a good interface!

Presentation Requirements

After you submit your assignment, your group will give a short presentation about it:

  • Introduce your system and give a brief overview of it.
  • Summarize the version of the system you have implemented. Pay particular attention to the how your system differs from the user manual and design document from the previous assignment.
  • Show us (briefly) the testing you did for your system.
  • Demonstrate your system. Show us the most interesting aspects of it, and how it all works together.
  • Your presentations will be timed: take no more than 7 minutes in total. Be prepared to answer questions afterwards both about your system, and how you team worked together. You should also be prepared to let audience members try out your system.
  • All team members are expected to attend the presentation, and over the three presentations for your project we want each team member to give at least one major part of the presentation.
  • Finally, the presentations for version 1 and version 2 are technical* presentations where you can assume the audience is made up entirely of software engineers. It is only in the final presentation of your completed project where you must be prepared for audience members who are not necessarily technically inclined.
